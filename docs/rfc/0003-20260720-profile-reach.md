# RFC 0003 — Profile reach (ticket 003a)

- **Date:** 2026-07-20
- **Ticket:** [003a — Profile reach](../tickets/003a-profile-reach.md)
- **Owning decisions:** [ADR-0007](../adr/0007-reach-is-an-inner-bound.md) · [ADR-0004](../adr/0004-producer-resolution-and-capability.md) · [ADR-0005](../adr/0005-build-time-composition.md)
- **Session:** [0014](../sessions/0014-20260720-003a-align-reach-placement.md)

Implementation plan for the build-time reach resolver. **No request-path change, no node behaviour
change** — the sole contract addition is the read-only `Provider.footprints` accessor.

## Scope

Deliver, for a woven profile, the per-parameter `Domain` it reaches — an **inner bound** (*every point
it names is servable*), for the surface to narrate and to author default windows. 003b consumes it.

**In scope:** one `Provider` accessor, one new module (`nodes/reach.py`), unit tests.
**Out of scope:** calling it (003b wires `compose()`), narration, window shaping, any `Capability`
change, any geometry primitive.

## Boundaries and ownership

| Boundary | Owner | Change |
|---|---|---|
| `Provider` — declares geometry | `nodes/providers/base.py` | **+`footprints` property** (abstract) |
| Open-Meteo leaf | `nodes/providers/open_meteo.py` | implement it off the existing `_build_capability` product |
| Reach resolution | **`nodes/reach.py`** (new) | `resolve_reach` + the `grid` rule |
| `Capability` | `manifold/capability.py` | **none** |
| `Weaver`, `Arbiter`, `Calculator`, `Reservoir`, `Store` | `nodes/` | **none** |
| `Domain` / `Interval` | `manifold/domain.py` | **none** — `intersect` stays a declared seam |

**Why `Provider` and not `Capability`:** reach is not a Manifold facet (ADR-0007) — putting an accessor
on `Capability` would force `UnionCapability` / `DerivedCapability` to implement something meaningless,
the vestigial-member problem ADR-0007 exists to remove. `Provider` already exposes `capability` and
`source_key` and is the thing that *declares* geometry.

**Why not from config:** `OfferingSpec` carries `name` / `parameters` / `store` and **no geometry** —
deliberately, per architecture.md ("geometry … stay off" the manifest). Footprints are built inside the
provider from its taps, cadence, and clock. So "build-time" means *runs at composition*, not *reads only
config*.

## Code shapes

### 1. The provider accessor

```python
# nodes/providers/base.py
class Provider(ABC):
    @property
    @abstractmethod
    def footprints(self) -> Mapping[ParameterId, Domain]:
        """Per-parameter declared geometry — the unit a profile's Reach is selected from (ADR-0007).

        Read at composition only; never on the request path. `capability` interprets the same
        declaration for admission.
        """
```

Open-Meteo returns what `_build_capability` already computed — the accessor exposes an existing fact,
it does not recompute one. Note the shape differs: `FootprintCapability.footprints` is
`{ParameterId: (ParameterDef, Domain)}`, so the accessor **projects** the `Domain` out of each pair.
The provider holds the `FootprintCapability` concretely, so no narrowing is needed at the call site.

### 2. Two build-time methods

Reach answers *how far*; a separate check answers *is it wired*. Both ship in this ticket, both walk
the same producer DAG over `ProfileDef`, and `compose()` runs the check first so reach can assume a
valid graph.

```python
# nodes/reach.py
def validate_calculators(profile: ProfileDef) -> None:
    """Every calculator's inputs must be producible. Raise CompositionError otherwise (ADR-0007)."""

def resolve_reach(profile: ProfileDef) -> Mapping[ParameterId, Domain]:
    """Per-parameter Reach — inner bound, selected never synthesized. Precondition: validated (ADR-0007)."""
```

**`validate_calculators`** — a calculator is an operator promise ("I derive wind"); if an input is
served by no producer, that promise is unkeepable and the build must say so, not fail silently at
runtime. Walks the DAG (memo + `visiting` cycle guard); raises `CompositionError` naming the
**calculator, the unserved input, and why** — or the cycle. Distinct from a *provider* parameter that
no enabled source serves: that degrades gracefully (absent, not an error) → v1 graceful-degrade.
The strict-vs-graceful interaction when a calculator input is served only by an *optional* provider is
[#35](../concerns.md#35-calculator-satisfiability-vs-optional-provider-degrade) (no v1 case — wind's
inputs come from the always-on keyless primary).

**`resolve_reach`** — three mutually recursive cases over the **one shape the graph already has**: a
Calculator's inputs are resolved by a scoped `Arbiter` ([weaver.py](../../src/meteoscape/nodes/weaver.py)),
so the same two sites repeat at every depth. Each `reach` call is parameterised by *which* producer set
it resolves over — all producers at the top, the calculator's scoped set below:

```
reach(producers, p)  = dominating([ footprint(prod, p) for prod in producers serving p ])
footprint(source, p) = source.provider.footprints[p]
footprint(calc, p)   = dominated([ reach(scoped_of(calc), i) for i in calc.inputs ])

resolve_reach(profile) = { p: reach(all_producers, p) for p in served parameters }
```

`scoped_of(calc)` mirrors `Weaver._weave_calculators`' `producers_for(reg.inputs)`, computed from the
registries rather than from built nodes: sources contribute their `provider.footprints` keys,
calculators contribute `reg.outputs`. Because `validate_calculators` ran first, every `reach(scoped, i)`
here has ≥1 candidate — reach's only raises are **geometry** (no dominating / no whole-box winner),
never wiring.

One principle — **the largest region the node can serve, as a `Domain` that already exists** — with the
quantifier supplied by the node type: an Arbiter serves where **any** producer does; a Calculator where
**all** its inputs do. Not a policy; a fact about the node. Two clarifications that fall out:

- **A Calculator competes at the top level like any producer.** If a future provider served
  `wind_speed` natively, its footprint and the Calculator's derived one meet in the same `dominating`
  selection — no special case.
- **`stored` is transparent to reach.** A `stored` calculator is wrapped in a `Reservoir` at weave;
  retention adds no capability, and the resolver reads registries, so the flag is never consulted.

### 3. The `grid` rule

```python
def _grid_dominating(candidates: Sequence[tuple[ProducerKey, Domain]]) -> Domain
def _grid_dominated(candidates: Sequence[tuple[ProducerKey, Domain]]) -> Domain
```

Both by **extent containment**, both raising `CompositionError` when unresolved — but with **different
procedures** (ADR-0007, The rule): `_grid_dominating` (Arbiter site) judges **X/Y first**, then the
remaining axes among X/Y ties; `_grid_dominated` (Calculator site) requires **whole-box containment in
every other candidate** — no axis ordering, since X/Y-first there would over-promise T the other
inputs cannot back. **On an equal-extent tie, return any candidate** (mutual containment ⇒ equal
extents ⇒ same promise); the choice is **unobservable** — in v1 the tied footprints even share the same
live axis objects (`_build_capability` builds X/Y/T once), so which one is returned changes nothing a
caller sees. No ordering or sort is needed, and none is imposed; a raise-on-non-unique would instead
fail every Calculator-bearing profile, since derived wind presents the tie on every parameter. A named
unit called directly — **no `ReachRule` protocol, no config, no registry**
([#28](../concerns.md#28-reconciler-interface-selection-ordering-vs-per-cell-fold)'s recorded lesson).

Containment is `Interval.contains` over each axis's `extent`. It must **not** reuse `Domain.matches`:
that is the request-side admission predicate, which `VantageAxis` specialises to *intersection*.

### 4. Errors

All `CompositionError` (existing, `nodes/composition.py`), all at build, in two kinds:

- **Wiring** (`validate_calculators`) — a calculator input no producer serves, or a calculator cycle;
  names the calculator + input.
- **Geometry** (`resolve_reach`) — no dominating producer (disjoint), or no whole-box winner among a
  calculator's inputs (sheared); names the conflicting producers by `SourceKey` / `CalculatorKey` and
  the axis. Possible because it works over `ProfileDef` — capabilities carry no producer identity.

## Flow

```
compose()                        [003b wires this]
  binders → ProfileDef
  weave(ProfileDef) → root Manifold
  validate_calculators(ProfileDef)          ← this RFC (wiring; raises before reach)
  resolve_reach(ProfileDef) → {ParameterId: Domain}   ← this RFC (geometry)
  Gateway(root) + surface(reach map)

resolve_reach   (precondition: validated)
  for each parameter across sources ∪ calculator outputs:
    collect candidate footprints (recursing through calculators, memoized)
    grid rule → winner's Domain (an existing object, live axes intact)
```

## Invariants

1. **No `Domain` is ever synthesized.** Every returned reach is a declared footprint object, so a
   clock-anchored `RollingAxis` inside it stays live — the T upper bound tracks the clock at read.
   *This is the invariant that makes 003b's default window correct;* synthesizing (e.g. by intersecting
   footprints) would freeze reach at build, and with v1's derived wind would degrade the default window
   to a single tick over uptime.
2. **Selection is fixed at build; values are not.** Which producer wins is decided once; its `Domain`
   stays live. Note the selection is itself **clock-dependent** — both site procedures compare
   `valid_time` extents, which a `RollingAxis` resolves against the clock — so where producers tie on
   X/Y and differ on T, the winner depends on boot time. Non-determinism, not incorrectness: every candidate is a real
   footprint, so whichever wins is a true inner bound. Unreachable in v1 (one provider cannot tie with
   itself), and **not** something to defend against with tie-break ordering — see Limitations.
3. **Nothing on the request path can read reach.** No node holds it.
4. `serves` and admission are byte-for-byte unchanged.

## Stages (TDD — red → green → refactor per stage)

**Stage 1 — `Provider.footprints`.**
Red: Open-Meteo exposes per-parameter footprints matching its capability declaration (global X/Y, tap Z,
rolling T). Green: abstract property + implementation. No behaviour change elsewhere; the existing
provider suite must stay green. While here, fix the pre-split `domain.py` docstrings that say "reach"
for a producer's footprint (module docstring, `ContinuousAxis`, `FootprintDomain`) — vocabulary only,
agreed session 0014.

**Stage 2 — the `grid` rule over a flat candidate list.**
Pure function, no `ProfileDef`. Red first, table-driven:

| Rule | Candidates | Expect |
|---|---|---|
| dominating | `{Europe × 16 d, Global × 10 d}` | `Global × 10 d` |
| dominating | `+ Arctic × 5 d` | `Global × 10 d` (unchanged) |
| dominating | `{Global × 16 d, Global × 10 d}` | `Global × 16 d` |
| dominating | `{Europe × 16 d, Americas × 10 d}` | `CompositionError`, naming both keys + axis |
| dominating | single candidate | itself |
| dominated | `{Europe × 10 d, Global × 16 d}` | `Europe × 10 d` (whole-box contained) |
| dominated | `{Europe × 10 d, Global × 5 d}` | `CompositionError` — **not** `Europe × 10 d` (sheared: X/Y-first would over-promise T) |
| either | two distinct equal-extent objects | **one of the inputs** (identity ∈ inputs), never a raise (the wind tie) |
| either | single candidate | itself |

Assert **object identity** on the winner — for the tie, that the result **is one of the inputs** (which
one is unobservable); for the rest, the specific winner. Identity, not equality, is what pins invariant
1 (nothing synthesized).

**Stage 3 — `resolve_reach` over sources only.**
Red: a two-source `ProfileDef` yields the dominating footprint per parameter; a parameter no source
serves is absent from the map. Also: reach **ignores priority** — `{Global × 10 d @1, Global × 16 d @2}`
→ 16 d.

**Stage 4 — `validate_calculators`.**
Red, all **without weaving**: a `ProfileDef` whose calculator input no producer serves raises
`CompositionError` naming the calculator + input; a cyclic `ProfileDef` raises naming the cycle; a
well-wired `ProfileDef` passes silently. This is the *wiring* method; owns the cycle guard.

**Stage 5 — Calculator recursion (reach).**
Red: a validated `ProfileDef` with derived wind yields reach for `wind_speed` / `wind_direction` whose
identity **is one of** the `wind_u` / `wind_v` footprints (the tie path — must not raise, must not
synthesize). A three-provider `{Europe×10 d, Global×5 d}`-shaped calculator input raises geometry
`CompositionError`. Memoization asserted by construction (a shared calculator resolved once).

**Stage 6 — liveness regression.**
The test that would have caught the defect this plan found: advance a fake `Clock`, re-read
`resolve_reach`'s result, assert the T upper bound **moved**. Guards invariant 1 against any future
reintroduction of synthesis.

## Failure handling

| Condition | Owner | Result |
|---|---|---|
| Calculator input no producer serves | `validate_calculators` | `CompositionError` — names calculator + input (broken promise; → [#35](../concerns.md#35-calculator-satisfiability-vs-optional-provider-degrade)) |
| Calculator cycle | `validate_calculators` | `CompositionError` — names the cycle |
| No producer dominates on X/Y (Arbiter site) | `resolve_reach` | `CompositionError` — naming producers + axis |
| No Calculator input whole-box contained in the others (crossing *or* sheared) | `resolve_reach` | `CompositionError`; **not** pre-decided beyond this (no v1 case) |
| Provider parameter no enabled source serves | — | absent from the map (graceful degrade, not an error) |

All at **startup**. Nothing reaches the request path.

## Migration, compatibility, rollout, observability

- **Migration:** none — new module, new accessor. No stored data, no wire format, no config schema.
- **Compatibility:** additive. The only breaking edge is third-party `Provider` implementations, of
  which there are none (Open-Meteo is the sole leaf).
- **Rollout:** inert until 003b calls it. Landing 003a alone changes no runtime behaviour.
- **Observability:** none added. The value is narrated by 003b; failures are startup-fatal and
  self-describing. Deliberately no logging in a pure build-time function.

## Limitations (accepted, in scope)

- **Calculator inputs with no whole-box winner raise** — crossing or sheared footprints — though their
  true intersection is computable and unique; never-synthesize forces the raise. No v1 Calculator hits
  it (wind's inputs are both Open-Meteo). The design for that case is deliberately **not** pre-decided.
- **Reach understates** where a regional producer reaches further than the dominating one — a Europe
  caller narrated 10 d may ask for and be served 16 d. Correct product statement, by ADR-0007.
- **Profile-level, not per-location.** Static; the per-location truth needs the deferred
  introspection tool.
- **The winner can depend on boot time** when producers tie on X/Y and differ on T (invariant 2).
  Deliberately **not** worked around here — a stable tie-break (e.g. by `ProducerKey`) would be
  arbitrary and would hide the condition rather than resolve it, and no v1 profile can produce a tie.
  If it ever becomes real, it is a reach-mode question, not a sort order.
