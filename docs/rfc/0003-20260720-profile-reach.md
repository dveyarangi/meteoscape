# RFC 0003 ‚Äî Profile reach (ticket 003a)

- **Date:** 2026-07-20
- **Ticket:** [003a ‚Äî Profile reach](../tickets/003a-profile-reach.md)
- **Owning decisions:** [ADR-0007](../adr/0007-reach-is-an-inner-bound.md) ¬∑ [ADR-0004](../adr/0004-producer-resolution-and-capability.md) ¬∑ [ADR-0005](../adr/0005-build-time-composition.md)
- **Session:** [0014](../sessions/0014-20260720-003a-align-reach-placement.md)

Implementation plan for the build-time reach resolver. **No request-path change, no node behaviour
change** ‚Äî the sole contract addition is the read-only `Provider.footprints` accessor.

## Scope

Deliver, for a woven profile, the per-parameter `Domain` it reaches ‚Äî an **inner bound** (*every point
it names is servable*), for the surface to narrate and to author default windows. 003b consumes it.

**In scope:** one `Provider` accessor, one new module (`nodes/reach.py`), unit tests.
**Out of scope:** calling it (003b wires `compose()`), narration, window shaping, any `Capability`
change, any geometry primitive.

## Boundaries and ownership

| Boundary | Owner | Change |
|---|---|---|
| `Provider` ‚Äî declares geometry | `nodes/providers/base.py` | **+`footprints` property** (abstract) |
| Open-Meteo leaf | `nodes/providers/open_meteo.py` | implement it off the existing `_build_capability` product |
| Reach resolution | **`nodes/reach.py`** (new) | `resolve_reach` + the `grid` rule |
| `Capability` | `manifold/capability.py` | **none** |
| `Weaver`, `Arbiter`, `Calculator`, `Reservoir`, `Store` | `nodes/` | **none** |
| `Domain` / `Interval` | `manifold/domain.py` | **none** ‚Äî `intersect` stays a declared seam |

**Why `Provider` and not `Capability`:** reach is not a Manifold facet (ADR-0007) ‚Äî putting an accessor
on `Capability` would force `UnionCapability` / `DerivedCapability` to implement something meaningless,
the vestigial-member problem ADR-0007 exists to remove. `Provider` already exposes `capability` and
`source_key` and is the thing that *declares* geometry.

**Why not from config:** `OfferingSpec` carries `name` / `parameters` / `store` and **no geometry** ‚Äî
deliberately, per architecture.md ("geometry ‚Ä¶ stay off" the manifest). Footprints are built inside the
provider from its taps, cadence, and clock. So "build-time" means *runs at composition*, not *reads only
config*.

## Code shapes

### 1. The provider accessor

```python
# nodes/providers/base.py
class Provider(ABC):
    @property
    @abstractmethod
    def footprints(self) -> Mapping[ParameterId, Domain]:
        """Per-parameter declared geometry ‚Äî the unit a profile's Reach is selected from (ADR-0007).

        Read at composition only; never on the request path. `capability` interprets the same
        declaration for admission.
        """
```

Open-Meteo returns what `_build_capability` already computed ‚Äî the accessor exposes an existing fact,
it does not recompute one. Note the shape differs: `FootprintCapability.footprints` is
`{ParameterId: (ParameterDef, Domain)}`, so the accessor **projects** the `Domain` out of each pair.
The provider holds the `FootprintCapability` concretely, so no narrowing is needed at the call site.

### 2. The resolver

```python
# nodes/reach.py
def resolve_reach(profile: ProfileDef) -> Mapping[ParameterId, Domain]:
    """Per-parameter Reach for a profile ‚Äî an inner bound, selected never synthesized (ADR-0007)."""
```

Three mutually recursive cases over the **one shape the graph already has** ‚Äî a Calculator's inputs are
resolved by a scoped `Arbiter` ([weaver.py](../../src/meteoscape/nodes/weaver.py)), so the same two
sites repeat at every depth. Each `reach` call is parameterised by *which* producer set it resolves
over ‚Äî all producers at the top, the calculator's scoped set below:

```
reach(producers, p)  = dominating([ footprint(prod, p) for prod in producers serving p ])
footprint(source, p) = source.provider.footprints[p]
footprint(calc, p)   = dominated([ reach(scoped_of(calc), i) for i in calc.inputs ])

resolve_reach(profile) = { p: reach(all_producers, p) for p in served parameters }
```

`scoped_of(calc)` mirrors `Weaver._weave_calculators`' `producers_for(reg.inputs)`, computed from the
registries rather than from built nodes: sources contribute their `provider.footprints` keys,
calculators contribute `reg.outputs`.

One principle ‚Äî **the largest region the node can serve, as a `Domain` that already exists** ‚Äî with the
quantifier supplied by the node type: an Arbiter serves where **any** producer does; a Calculator where
**all** its inputs do. Not a policy; a fact about the node.

### 3. The `grid` rule

```python
def _grid_dominating(candidates: Sequence[tuple[ProducerKey, Domain]]) -> Domain
def _grid_dominated(candidates: Sequence[tuple[ProducerKey, Domain]]) -> Domain
```

Both by **extent containment**, both raising `CompositionError` when unresolved ‚Äî but with **different
procedures** (ADR-0007, The rule): `_grid_dominating` (Arbiter site) judges **X/Y first**, then the
remaining axes among X/Y ties; `_grid_dominated` (Calculator site) requires **whole-box containment in
every other candidate** ‚Äî no axis ordering, since X/Y-first there would over-promise T the other
inputs cannot back. **Equal-extent ties resolve to the first candidate** at both sites (mutual
containment ‚áí equal extents ‚áí same promise); this is load-bearing, not a nicety ‚Äî derived wind's
`wind_u` / `wind_v` footprints are distinct equal-extent objects, so a raise-on-non-unique would fail
every Calculator-bearing profile. A named unit called directly ‚Äî **no `ReachRule` protocol, no config,
no registry** ([#28](../concerns.md#28-reconciler-interface-selection-ordering-vs-per-cell-fold)'s
recorded lesson).

Containment is `Interval.contains` over each axis's `extent`. It must **not** reuse `Domain.matches`:
that is the request-side admission predicate, which `VantageAxis` specialises to *intersection*.

### 4. Errors

`CompositionError` (existing, `nodes/composition.py`), naming the conflicting producers by
`SourceKey` / `CalculatorKey` and the axis on which dominance failed. Being over `ProfileDef` is what
makes that possible ‚Äî capabilities carry no producer identity.

## Flow

```
compose()                     [003b wires this]
  binders ‚Üí ProfileDef
  weave(ProfileDef) ‚Üí root Manifold
  resolve_reach(ProfileDef) ‚Üí {ParameterId: Domain}     ‚Üê this RFC
  Gateway(root) + surface(reach map)

resolve_reach
  for each parameter across sources ‚à™ calculator outputs:
    collect candidate footprints (recursing through calculators, memoized)
    grid rule ‚Üí winner's Domain (an existing object, live axes intact)
```

## Invariants

1. **No `Domain` is ever synthesized.** Every returned reach is a declared footprint object, so a
   clock-anchored `RollingAxis` inside it stays live ‚Äî the T upper bound tracks the clock at read.
   *This is the invariant that makes 003b's default window correct;* synthesizing (e.g. by intersecting
   footprints) would freeze reach at build, and with v1's derived wind would degrade the default window
   to a single tick over uptime.
2. **Selection is fixed at build; values are not.** Which producer wins is decided once; its `Domain`
   stays live. Note the selection is itself **clock-dependent** ‚Äî both site procedures compare
   `valid_time` extents, which a `RollingAxis` resolves against the clock ‚Äî so where producers tie on
   X/Y and differ on T, the winner depends on boot time. Non-determinism, not incorrectness: every candidate is a real
   footprint, so whichever wins is a true inner bound. Unreachable in v1 (one provider cannot tie with
   itself), and **not** something to defend against with tie-break ordering ‚Äî see Limitations.
3. **Nothing on the request path can read reach.** No node holds it.
4. `serves` and admission are byte-for-byte unchanged.

## Stages (TDD ‚Äî red ‚Üí green ‚Üí refactor per stage)

**Stage 1 ‚Äî `Provider.footprints`.**
Red: Open-Meteo exposes per-parameter footprints matching its capability declaration (global X/Y, tap Z,
rolling T). Green: abstract property + implementation. No behaviour change elsewhere; the existing
provider suite must stay green. While here, fix the pre-split docstrings in the touched files that say
"reach" for a producer's footprint (`domain.py` "a continuous provider reach" / `ContinuousAxis` "X/Y
reach"; `base.py` module docstring) ‚Äî vocabulary only, agreed session 0014.

**Stage 2 ‚Äî the `grid` rule over a flat candidate list.**
Pure function, no `ProfileDef`. Red first, table-driven:

| Rule | Candidates | Expect |
|---|---|---|
| dominating | `{Europe √ó 16 d, Global √ó 10 d}` | `Global √ó 10 d` |
| dominating | `+ Arctic √ó 5 d` | `Global √ó 10 d` (unchanged) |
| dominating | `{Global √ó 16 d, Global √ó 10 d}` | `Global √ó 16 d` |
| dominating | `{Europe √ó 16 d, Americas √ó 10 d}` | `CompositionError`, naming both keys + axis |
| dominating | single candidate | itself |
| dominated | `{Europe √ó 10 d, Global √ó 16 d}` | `Europe √ó 10 d` (whole-box contained) |
| dominated | `{Europe √ó 10 d, Global √ó 5 d}` | `CompositionError` ‚Äî **not** `Europe √ó 10 d` (sheared: X/Y-first would over-promise T) |
| dominated | two distinct equal-extent objects | the **first**, by identity ‚Äî never a raise (the wind case) |
| dominating | two distinct equal-extent objects | the **first**, by identity |
| dominated | single candidate | itself |

Assert **object identity** on the winner, not equality ‚Äî that is what pins invariant 1.

**Stage 3 ‚Äî `resolve_reach` over sources only.**
Red: a two-source `ProfileDef` yields the dominating footprint per parameter; a parameter no source
serves is absent from the map. Also: reach **ignores priority** ‚Äî `{Global √ó 10 d @1, Global √ó 16 d @2}`
‚Üí 16 d.

**Stage 4 ‚Äî Calculator recursion + cycle guard.**
Red: a `ProfileDef` with derived wind yields reach for `wind_speed` / `wind_direction` identical (by
object identity) to the **first input's** footprint ‚Äî `wind_u` / `wind_v` footprints are distinct
equal-extent objects, so this is the tie path, and it must **not** raise; a cyclic `ProfileDef` raises
`CompositionError` **without weaving**. Memoization asserted by construction (a shared calculator
resolved once).

**Stage 5 ‚Äî liveness regression.**
The test that would have caught the defect this plan found: advance a fake `Clock`, re-read
`resolve_reach`'s result, assert the T upper bound **moved**. Guards invariant 1 against any future
reintroduction of synthesis.

## Failure handling

| Condition | Result |
|---|---|
| No producer dominates on X/Y (Arbiter site) | `CompositionError` ‚Äî naming producers + axis |
| No Calculator input whole-box contained in the others (crossing *or* sheared) | `CompositionError` ‚Äî same shape; **not** pre-decided beyond this (no v1 case) |
| Calculator cycle | `CompositionError` ‚Äî from the resolver's own guard |
| Parameter served by nobody | absent from the map (not an error) |

All at **startup**. Nothing reaches the request path.

## Migration, compatibility, rollout, observability

- **Migration:** none ‚Äî new module, new accessor. No stored data, no wire format, no config schema.
- **Compatibility:** additive. The only breaking edge is third-party `Provider` implementations, of
  which there are none (Open-Meteo is the sole leaf).
- **Rollout:** inert until 003b calls it. Landing 003a alone changes no runtime behaviour.
- **Observability:** none added. The value is narrated by 003b; failures are startup-fatal and
  self-describing. Deliberately no logging in a pure build-time function.

## Limitations (accepted, in scope)

- **Calculator inputs with no whole-box winner raise** ‚Äî crossing or sheared footprints ‚Äî though their
  true intersection is computable and unique; never-synthesize forces the raise. No v1 Calculator hits
  it (wind's inputs are both Open-Meteo). The design for that case is deliberately **not** pre-decided.
- **Reach understates** where a regional producer reaches further than the dominating one ‚Äî a Europe
  caller narrated 10 d may ask for and be served 16 d. Correct product statement, by ADR-0007.
- **Profile-level, not per-location.** Static; the per-location truth needs the deferred
  introspection tool.
- **The winner can depend on boot time** when producers tie on X/Y and differ on T (invariant 2).
  Deliberately **not** worked around here ‚Äî a stable tie-break (e.g. by `ProducerKey`) would be
  arbitrary and would hide the condition rather than resolve it, and no v1 profile can produce a tie.
  If it ever becomes real, it is a reach-mode question, not a sort order.
